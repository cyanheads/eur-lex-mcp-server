# 0.9.6 — 2026-07-16

## Added

- **`eurlex_query_sparql` result-cap disclosure** ([#52](https://github.com/cyanheads/eur-lex-mcp-server/issues/52)) — an `enrichment` block (`truncated`, `shown`, `cap`) now fires via `ctx.enrich.truncated()` when the `maxSparqlResults` ceiling both clamped the query and filled the result, so a `content[]`-only client can tell a genuinely complete 100-row result from one truncated with more rows upstream. `CellarSparqlService.queryWithVars` surfaces a new `limitEnforced` flag; disclosure gates on `limitEnforced && bindings.length === maxResults` — exact equality, since only the first `LIMIT` is rewritten and a subselect can leave the outer query uncapped.

## Fixed

- **`eurlex_query_sparql` text output renders full SPARQL terms** ([#54](https://github.com/cyanheads/eur-lex-mcp-server/issues/54)) — `format()` now emits each cell in SPARQL/Turtle syntax (`<iri>`, `_:bnode`, `"lexical"`, `"lexical"@en`, `"lexical"^^xsd:date`) instead of the bare `value`, so `content[]` carries the same `type`/`datatype`/`xml:lang` metadata as `structuredContent`. XSD datatypes abbreviate to the auto-injected `xsd:` prefix; the full row set restored in #50 is unchanged.

## Security

Caller-supplied input reaching a CELLAR SPARQL query is now escaped or validated uniformly. The existing quote escaping already blocked literal breakout, so these harden a disclosure surface — a malformed literal or IRI leaked Virtuoso's raw `SP030` compiler error, with the internal query text and PREFIX block attached, in place of a tool's own error — rather than an injection vector.

- **`escapeSparqlLiteral` escapes control characters** ([#53](https://github.com/cyanheads/eur-lex-mcp-server/issues/53)) — now escapes `\n`/`\r`/`\t` after the backslash and quote passes (order is load-bearing), so an interior newline in `celex_number` or `eli_uri` no longer makes the short literal unparseable.
- **`isSafeSparqlIri` backs every URI input** ([#53](https://github.com/cyanheads/eur-lex-mcp-server/issues/53), [#60](https://github.com/cyanheads/eur-lex-mcp-server/issues/60)) — a shared predicate requiring an `http` prefix and rejecting `/[\s<>"]/` now guards `eurlex_get_document.work_uri`, `eurlex_get_relations.work_uri`, and `eurlex_search_documents.eurovoc_concept`, replacing three divergent hand-rolled guards that admitted tabs, newlines, and — for `eurovoc_concept` — an opening `<`.
- **Resource escaping routes through the shared helper** ([#61](https://github.com/cyanheads/eur-lex-mcp-server/issues/61)) — `eurlex://document/{celexNumber}` and `eurlex://document/{celexNumber}/relations` now call `escapeSparqlLiteral` instead of a quote-only pass, so a CELEX ending in `\` no longer escapes the closing quote.
- **Tool keyword escaping routes through the shared helper** ([#62](https://github.com/cyanheads/eur-lex-mcp-server/issues/62)) — the four remaining hand-rolled sites in `eurlex_browse_subjects` (`keyword`), `eurlex_get_cases` (`case_number`, `keyword`), and `eurlex_search_documents` (`keyword`) now call `escapeSparqlLiteral`, escaping last so the preceding `.trim()` still strips a trailing tab. Every CELLAR query builder now routes literal escaping through one helper.
