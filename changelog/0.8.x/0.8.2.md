# 0.8.2 — 2026-07-03

## Added

- **`eurlex_get_cases` case titles parsed into structured fields** ([#40](https://github.com/cyanheads/eur-lex-mcp-server/issues/40)) — the raw CELLAR `#`-delimited expression title (e.g. `Judgment of the Court…#WhatsApp Ireland Ltd v European Data Protection Board.#Appeal – …#Case C-97/23 P.`) is now parsed via `parseCaseLawTitle()` into `display_title`, `parties`, `subject_matter`, and `case_reference` output fields. The raw `title` field is preserved unchanged.

## Fixed

- **`eurlex_get_cases` `case_type` filter tests the CDM resource-type, not a CELEX substring** ([#38](https://github.com/cyanheads/eur-lex-mcp-server/issues/38)) — the old `FILTER(CONTAINS(STR(?celexNumber), "CJ"))`-style test also matched abstract (`_RES` → `ABSTRACT_JUR`) and summary (`_SUM` → `SUM_JUR`) sibling works, which carry the parent judgment's CELEX type letters under a distinct CELEX. The filter now requires `?work cdm:work_has_resource-type <…/JUDG|ORDER|OPIN_AG>`, excluding the siblings.
- **`parseCaseLawTitle()` handles CELLAR's plural joined-case reference form** ([#42](https://github.com/cyanheads/eur-lex-mcp-server/issues/42)) — the case-reference anchor widened from `/^Case\b/i` to `/^Cases?\b/i` (`\b` never asserts between "Case" and a trailing "s"), so titles like `Cases T-318/24 and T-362/24.` now resolve `case_reference` correctly instead of leaving it unset and misattributing the case list to `subject_matter`.
