# 0.4.1 — 2026-06-30

## Fixed

- **`eurlex_get_relations` `amended_by` / `consolidated_version`** ([#19](https://github.com/cyanheads/eur-lex-mcp-server/issues/19)) — both mapped to CDM predicates that carry zero triples in CELLAR (`cdm:resource_legal_amended_by_resource_legal`, `cdm:resource_legal_has_consolidated_version`), so requesting either always failed with `no_relations` even for heavily-amended acts. `amended_by` now traverses the incoming side of `cdm:resource_legal_amends_resource_legal`; `consolidated_version` traverses incoming `cdm:act_consolidated_consolidates_resource_legal`. `amends` and `legal_basis` are now outgoing-only (their incoming rows are semantically `amended_by` and another work's `legal_basis`). Each relation type is queried independently with its own `LIMIT`, run concurrently, so a high-volume type like `cites` no longer starves rarer types under a single shared cap. The `eurlex://document/{celexNumber}/relations` resource carried the identical predicate map and is fixed by the same shared traversal.
- **`in_force` on `eurlex_get_document` and the `eurlex://document/{celexNumber}` resource** ([#20](https://github.com/cyanheads/eur-lex-mcp-server/issues/20)) — always reported `false`. Virtuoso serializes `cdm:resource_legal_in-force` as an `xsd:boolean` whose SPARQL-JSON lexical is `"1"` / `"0"`, but the handler tested `=== 'true'`. New `CellarSparqlService.parseBoolean()` accepts both `"true"` / `"1"` and `"false"` / `"0"`, and returns `undefined` for an absent or unrecognized value rather than coercing to a false negative.
- **`eurlex_lookup_celex` `found: false`** ([#22](https://github.com/cyanheads/eur-lex-mcp-server/issues/22)) — a well-formed but nonexistent identifier threw `not_found`, making the documented `found: false` outcome unreachable and defeating the tool's validate-before-fetch role. A well-formed identifier that resolves to no work now returns `{ found: false }`; malformed or undetectable input still raises `ambiguous_identifier`.
- **`eurlex_query_sparql` projected variables** ([#23](https://github.com/cyanheads/eur-lex-mcp-server/issues/23)) — empty result sets returned `variables: []` because the projection was derived from the first binding's keys. New `CellarSparqlService.queryWithVars()` reports the SPARQL 1.1 `head.vars` projection independent of binding count. The tool description now notes CELEX is stored as an `xsd:string`-typed literal — match it with `FILTER(STR(?celex) = "…")`; a plain `FILTER(?celex = "…")` matches nothing.
